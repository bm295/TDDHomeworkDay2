# Use Cases

## Purpose
This document defines the application-layer use cases that AI agents should implement against. It turns the product requirements into explicit commands, queries, inputs, outputs, and error contracts.

## Scope
- Library-first delivery
- No UI/API layer required yet
- File-backed repositories are allowed through infrastructure abstractions
- Authentication and payment remain out of scope

## Actors
- `StoreAdmin`: maintains catalog, pricing inputs, stock, and reports
- `Cashier`: builds carts and checks out customer orders
- `SystemIntegrator` (future): imports and exports data

## Canonical Use Cases

### UC-01 RegisterBook
Purpose: add a new sellable title to the catalog.

Input:
- `Isbn`
- `Title`
- `Author`
- `Category`
- `BasePrice`

Rules:
- `Book.Id` is system-generated by the application layer or repository, not supplied by the user.
- `Isbn` must be unique.
- `Title` and `Author` are required.
- `Category` is free text in v1.
- `BasePrice` must be greater than or equal to zero.
- New books default to `IsActive = true`.

Success result:
- Persisted `Book`
- Generated `BookId`

Failure result:
- `DomainValidationException` for invalid field values
- `DuplicateEntityException` for duplicate ISBN

### UC-02 UpdateBookMetadata
Purpose: change book metadata without changing historical order data.

Input:
- `BookId`
- Updated metadata fields

Rules:
- Unknown `BookId` is rejected.
- ISBN uniqueness still applies.
- Historical orders keep their existing price snapshots.

Success result:
- Updated `Book`

### UC-03 DeactivateBook
Purpose: stop a title from being newly sold without deleting historical references.

Rules:
- Deactivation sets `IsActive = false`.
- Deactivated books remain readable from the catalog.
- Existing orders and reports must continue to reference the book.
- Hard delete is out of scope for v1.

### UC-04 AdjustStock
Purpose: increase or decrease available inventory for one title.

Input:
- `BookId`
- `QuantityDelta`
- optional reason string for audit/logging later

Rules:
- The book must exist.
- Negative results are not allowed.
- `ReorderThreshold` is managed on `InventoryItem`, not on each adjustment command.
- After adjustment, low-stock state is `QuantityOnHand <= ReorderThreshold`.

Success result:
- Updated `InventoryItem`

### UC-05 AddCartLine
Purpose: add a book to a transient cart or increase an existing line quantity.

Input:
- `BookId`
- `Quantity`

Rules:
- `Quantity` must be greater than zero.
- Adding the same `BookId` multiple times merges into one logical cart line.
- `UnitPrice` is copied from the current book base price when the line is first created.
- Adding more quantity to an existing line keeps the existing unit-price snapshot in v1.
- Pricing does not reserve stock.

### UC-06 RemoveCartLine
Purpose: reduce quantity or remove a line from a cart.

Rules:
- Removing a quantity down to zero deletes the line.
- Removing a missing line is an error.

### UC-07 PriceCart
Purpose: calculate cart totals without mutating inventory.

Output:
- `Subtotal`
- `DiscountTotal`
- `GrandTotal`
- optional pricing trace entries

Rules:
- Pricing behavior must follow `PRICING_SPEC.md`.
- Pricing must be deterministic for the same cart contents.

### UC-08 Checkout
Purpose: convert a cart into an order and decrement inventory atomically at the application level.

Preconditions:
- Cart is not empty.
- Every referenced book exists and is active.
- Sufficient stock exists for every cart line.

Success result:
- Created `Order`
- Inventory decremented for all lines
- `Order.CreatedAtUtc` recorded in UTC

Failure result:
- No partial inventory changes are committed if checkout fails
- `InsufficientStockException` if any line exceeds stock

### UC-09 GetInventorySnapshot
Purpose: read current stock and low-stock state for operators.

Output per row:
- `BookId`
- `Title`
- `QuantityOnHand`
- `ReorderThreshold`
- `IsLowStock`

### UC-10 GetDailySalesSummary
Purpose: provide daily reporting from stored orders.

Output:
- Order count
- Units sold
- Gross sales
- Discount total
- Net sales

Rules:
- Report groups by UTC calendar day in v1.
- Empty day returns zero totals, not an error.

## Out of Scope
- Shopping sessions shared across devices
- Partial checkout
- Payment authorization
- Stock reservation before checkout
- Multi-currency pricing
